# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

A hyper-local service marketplace (like armut.com) for Didim, Aydın, connecting customers with local service providers ("usta"). Flutter client (mobile/web/desktop) backed by Supabase (Postgres + Auth + PostgREST).

## Commands

```bash
flutter pub get                      # install dependencies
flutter analyze                      # static analysis — must be clean before considering work done
flutter test                         # run all tests (currently one file: test/widget_test.dart)
flutter test test/widget_test.dart   # run the single test file

flutter run -d chrome                # run interactively (opens a real browser window — does not
                                      # work from a non-interactive/headless shell; use web-server instead, see below)
flutter run -d web-server --web-port=<port> --web-hostname=127.0.0.1   # headless dev server, open the URL manually
flutter build web --release          # release web build (build/web) — much faster/more reliable to
                                      # automate against than the debug web-server, which uses DDC and
                                      # can take 60s+ to compile on first load
flutter build apk --release          # release Android APK -> build/app/outputs/flutter-apk/app-release.apk

supabase db push --yes               # apply new migrations in supabase/migrations/ to the linked remote project
supabase login --token <token>       # CLI account-level auth (needed for db push / projects create / etc.)
```

No CI config exists in this repo. `flutter analyze` + `flutter test` are the only automated gates.

### Windows/Gradle gotcha
Android release builds on this machine can fail with `Could not read workspace metadata from ...\.gradle\caches\<ver>\transforms\...\metadata.bin` — a corrupted global Gradle cache unrelated to the project. Fix: `cd android && ./gradlew --stop` (must kill the daemon first, or the cache files are locked), then delete `~/.gradle/caches/<version>/` (or just its `transforms` subdir) and rebuild.

### `supabase db push` connection gotcha
`supabase login --token <token>` only authenticates the CLI against the Supabase *management* API — `db push`/`migration repair` additionally need a direct Postgres connection and will fail with `unexpected login role status 401` until `SUPABASE_DB_PASSWORD` is also set (or `-p <password>` passed), which is a *separate* credential (Project Settings → Database → database password, resettable but not re-viewable) from the access token. In this environment the pooler host (`aws-0-<region>.pooler.supabase.com`) has also been observed to reject a freshly-reset password outright (`password authentication failed`, SQLSTATE 28P01) regardless of retries/propagation delay, and the direct host (`db.<ref>.supabase.co`) doesn't resolve for this project (no IPv4 add-on). When the CLI path doesn't cooperate, the reliable fallback is pasting the migration SQL directly into the Supabase Dashboard's **SQL Editor** and running it there — just double-check the dashboard is on the right project first (`Project Settings → General → Reference ID` should match the `xkdmgrgxjrlwuztzkyuq` ref baked into `lib/core/config/supabase_config.dart`'s URL; running it against the wrong project surfaces as `relation "public.service_requests" does not exist`). Note this leaves the CLI's own remote migration-history table out of sync — a later `supabase db push` won't know the migration was already applied.

### GitHub Pages deployment
The web build is deployed to `https://ercinnn.github.io/usta_didim/` from the `gh-pages` branch (root, plain static files — no GitHub Actions workflow). To redeploy after a change:
```bash
flutter build web --release --base-href /usta_didim/    # base-href must match the Pages subpath, or asset URLs 404
```
Then copy `build/web/*` (plus a touched `.nojekyll` file, so Pages doesn't ignore Flutter's `_`-prefixed asset paths) onto the `gh-pages` branch and push. Since `gh-pages` has unrelated history from `master`, do this in a throwaway `git worktree` rather than switching branches in the main working copy: `git worktree add <tmp-dir> gh-pages`, clear its tracked files, copy the new `build/web` output in, commit, push, then `git worktree remove --force <tmp-dir>`.

## Architecture

**Feature-first**, not layer-first: `lib/features/{auth,profile,requests,offers,reviews}/{data,domain,presentation}`. Cross-feature imports are normal and expected (e.g. `requests` reads the provider's category from `profile`, `offers` composes both `requests` and `profile` data) — don't try to fully decouple features.

- `lib/core/config/supabase_config.dart` — Supabase URL + publishable key (safe to expose client-side, protected by RLS). Never put the `service_role` key or a personal access token in client code.
- `lib/core/constants/` — shared constants used across features: `didimNeighborhoods` (strict 12-item list) and `serviceCategories`.
- `lib/core/theme/` — the app's design system: a Glassmorphism / "Liquid Glass" identity (`GlassColors` — blue primary/accent, light+dark background gradients, translucent glass fill/border tokens; `GlassSpacing` — 8pt grid, corner radii, motion constants; `GlassTheme.light()`/`.dark()` — the Material 3 `ThemeData`, Inter throughout; `AppTextStyles.mono` — JetBrains Mono for prices/category tags). Applied once in `main.dart`'s `MaterialApp.theme`/`darkTheme`/`themeMode`; screens should pull colors from `Theme.of(context)`/`GlassColors` rather than hardcoding Material defaults.
- `lib/core/widgets/` — shared frosted-glass widgets, all built on `BackdropFilter`: `GlassContainer` (base blurred/translucent surface), `GlassScaffold` (gradient background + width-constrained/centered body for web), `GlassAppBar`, `GlassButton` (gradient-filled primary / translucent secondary), `GlassTextField`, `GlassServiceCard` (the card used for every request/offer/job list item — takes an `eyebrow` category label, `accentColor` for status, optional `trailing`), `VerifiedStamp` (badge shown next to a provider's name when `providers.is_verified`), `StarRating` (5-star row, interactive when given `onChanged`, read-only otherwise).
- Each feature follows `data/` (Supabase-backed repository, one per table), `domain/` (plain model classes with a `fromMap` factory matching the Postgres row shape, plus enum-like status classes with a `fromDb`/`.name` mapping to the DB's `text` check-constraint values), `presentation/` (Riverpod providers + screens).

**State management**: Riverpod 3.x, no code generation — plain `Provider`/`FutureProvider`/`FutureProvider.family`/`StreamProvider`. Note the Riverpod 3 API change: `AsyncValue.valueOrNull` was renamed to `.value`.

**Auth/navigation gating** (`lib/features/auth/presentation/auth_gate.dart` → `lib/features/profile/presentation/profile_gate.dart`): `MaterialApp.home` is `AuthGate`, which watches `authStateChangesProvider` and swaps between `LoginScreen` and `ProfileGate`. `ProfileGate` watches `currentProfileProvider`; if the profile's role is `provider`, it further gates on `currentProviderProfileProvider` to decide between `ProviderProfileScreen` (profile not completed yet) and `ProviderHomeScreen`. Customers go straight to `CustomerHomeScreen`.

Important: any screen that gets to this gate via `Navigator.push` (e.g. `SignUpScreen`, `ProviderProfileScreen`) must explicitly `Navigator.pop`/`popUntil` back to the root after finishing its action and invalidating the relevant provider. The gate widgets underneath do rebuild reactively, but if a pushed route is still on top of the Navigator stack, the user keeps seeing the pushed screen — this was a real bug (found via E2E testing) where sign-up completed successfully but the UI never advanced because nothing popped `SignUpScreen`.

### Supabase backend

Schema + RLS policies live in `supabase/migrations/`, applied via `supabase db push` (see the CLI gotcha above if that fails). Eight tables: `profiles` (1:1 with `auth.users`, has `role`: customer/provider), `providers` (1:1 with `profiles` for provider accounts, has `category`/`neighborhood`/`is_verified`/`rating`), `service_requests` (customer's job postings, `status`: open/pending/completed/cancelled), `offers` (provider bids on requests), `reviews` (customer's 1-5 star rating + optional comment on a `completed` request, one per request via a `unique` constraint on `request_id`), `messages` (per-request chat, opens once an offer is `accepted`), `notifications` (in-app feed, see below), `device_tokens` (FCM tokens for push, see "Push notifications" below).

`providers.rating` is never written directly by the app — a trigger (`refresh_provider_rating`, added in `20260707120000_add_reviews.sql`) recomputes it as the average of that provider's `reviews` rows on every insert. Don't add app-side code that writes `providers.rating`; add a review row instead and let the trigger do it. The customer-facing flow that produces a review is in `request_detail_screen.dart`: once a request's accepted offer exists and the customer marks it `completed` (`ServiceRequestRepository.markCompleted`), a review form replaces the "mark completed" button; once submitted it flips to a read-only summary (enforced server-side too, via the `unique(request_id)` constraint plus an RLS insert policy requiring `status = 'completed'` and an `accepted` offer from that exact provider).

`notifications` rows follow the same pattern: the app never inserts into `notifications` directly (there's no insert RLS policy for regular users at all) — four `SECURITY DEFINER` triggers in `20260708000000_add_messages_and_notifications.sql` write them server-side on `offers` insert/update, `messages` insert, and `service_requests` update-to-completed, because the recipient of a notification is usually not the actor who caused it (e.g. a provider's offer insert must write a notification row for the *customer*, who never granted the provider insert access to their own `notifications`). `lib/features/notifications/` reads this table via a realtime `.stream()` (Riverpod `StreamProvider`, the same pattern as `messages` in `lib/features/messages/` and `authStateChangesProvider`).

**Messaging RLS**: unlike the `providers`/`offers` cycle, `messages` policies (`sender is the request's customer OR the provider with an accepted offer on it`) don't need the `SECURITY DEFINER` bypass — `messages` is a new leaf table that `offers`/`service_requests` policies never reference back, so there's no cycle to break (same reasoning as `reviews`).

**Realtime publication gotcha**: creating a table does *not* automatically make `.stream()` (Riverpod `StreamProvider`, used by `messages`/`notifications`) receive live updates — Supabase Realtime's `postgres_changes` only fires for tables explicitly added to the `supabase_realtime` publication (normally a Dashboard → Database → Replication toggle per table). This was missed when `messages`/`notifications` were first added; `20260708000200_enable_realtime_messages_notifications.sql` (`alter publication supabase_realtime add table public.messages/notifications;`) fixes it. Any *future* table that needs `.stream()` needs the same migration line, or it'll silently just never push updates (no error — the initial fetch works fine, only live changes don't arrive) — confirmed working end-to-end via the two-account Playwright run described below (bell badge and chat messages appearing on the other side without a page reload).

### Push notifications
In-app notifications (the `notifications` table + bell icon) work standalone with no external dependency. Actual push delivery additionally needs a Firebase project, which does **not exist yet** in this repo — `firebase_core`/`firebase_messaging` are in `pubspec.yaml` and `main.dart` calls `Firebase.initializeApp()` inside a `try/catch` that silently no-ops until it's configured, specifically so the rest of the app (including in-app notifications) keeps working without it. `lib/features/notifications/data/push_token_repository.dart` registers the device's FCM token into `device_tokens` (Android + web only, matching the platforms this app actually ships to per this file) after `Firebase.initializeApp()` succeeds. To turn push on: create a Firebase project, drop `google-services.json` into `android/app/`, run `flutterfire configure` (or hand-write `firebase_options.dart`) to make `Firebase.initializeApp()` succeed, generate a web VAPID key (pass via `--dart-define=FCM_VAPID_KEY=...` at build time), and deploy `supabase/functions/send-push-notification/` with `FIREBASE_PROJECT_ID`/`FIREBASE_SERVICE_ACCOUNT` secrets set plus a Database Webhook on `notifications` insert to invoke it.

**RLS recursion gotcha**: a policy on table A that subqueries table B, where table B has a policy that subqueries table A (directly or transitively), makes Postgres reject it with `infinite recursion detected in policy for relation "..."` (42P17) at query time — regardless of whether the specific row-matching branch would ever actually be reached. This bit us twice in this schema (`providers` ↔ `offers` ↔ `service_requests` cycles). Fix pattern: wrap the cross-table check in a `SECURITY DEFINER` SQL function (owned by a role with `BYPASSRLS`, e.g. the migration-applying `postgres` role) so its internal query doesn't re-trigger RLS evaluation on the other table. See `20260706000100_fix_providers_offers_rls_recursion.sql` and `20260706000200_providers_see_requests_via_own_offers.sql` for the pattern (`provider_has_offer_on_customer_request`, `request_has_offer_from_provider`).

Nested/embedded PostgREST selects (e.g. `offers.select('*, providers(business_name, rating), service_requests(title, ...)')`) enforce RLS on the embedded table too — a row can be visible on the base table but its embedded fields come back null if the current user can't independently pass the embedded table's RLS. `lib/features/offers/domain/offer.dart` handles this by making the joined fields nullable.

### Testing against the real backend (E2E, not committed)

No E2E test suite is checked into the repo — verification during development was done ad hoc with Playwright (`npm install playwright` + `npx playwright install chromium` in a scratch dir) driving `flutter build web --release` served statically (e.g. `python -m http.server` from a directory containing a `usta_didim/` subfolder with the build output, so the `--base-href /usta_didim/` asset paths resolve the same way they do on GitHub Pages). If doing this again:
- Flutter web (CanvasKit) paints to canvas; there's no real DOM text. The accessibility tree only populates real DOM nodes (`aria-label` on text field inputs, `role="button"`/`role="radio"` with visible text content on tappable widgets) after `flt-semantics-placeholder` is activated — but that element is a 1×1px node positioned at `(-1, -1)`, so a normal `page.click(...)` or `page.mouse.click(5, 5)` either times out ("element is outside of the viewport") or silently hits nothing. Use `page.locator('flt-semantics-placeholder').dispatchEvent('click')` instead, once per page load.
- Even with semantics enabled, `.fill()` on a `TextFormField` can silently no-op for some `keyboardType`s (observed on `TextInputType.phone`) — the locator resolves and no error is thrown, but the value never lands. The reliable pattern is `locator.click()` then `page.keyboard.type(text, { delay: 20 })` (a small per-key delay also avoids the first 1-2 characters getting eaten by a focus/render race right after the click).
- `DropdownButtonFormField` menu overlays and non-selected `TabBar` tab labels never attach to the semantics/DOM tree at all (canvas-only), even with accessibility enabled. They can't be targeted by any DOM selector — only raw coordinate clicks (`page.mouse.click(x, y)`) work. The empty field's `labelText` (unselected) *does* attach as locatable text (`page.getByText('Hizmet Kategorisi')`), so it's reliable for getting the trigger's bounding box. Click-to-open then click-same-position-again to pick the topmost item only holds for short lists (worked for a 4-item category dropdown); a 12-item neighborhood dropdown opened *centered on screen* instead and the second click landed on item ~5, not item 1 — screenshot after opening to confirm the actual item positions for anything but very short lists, rather than assuming the trigger's on-screen position.
- `flutter run -d chrome` reliably fails to launch a real browser window from a non-interactive shell in this environment ("Failed to launch browser after 3 tries") — use `-d web-server` and open the URL in an actual browser session instead.
- A widget that visually groups several `Text` children under one `InkWell`/tappable region (e.g. `GlassServiceCard`'s eyebrow + title + subtitle) gets merged into a *single* semantics node whose accessible name is the concatenation of all of them — `page.getByText('exact title', { exact: true })` then never matches (only the merged string does), even though the title is clearly visible. Drop `exact: true` (substring match) for anything inside a tappable card; reserve exact matching for standalone labels/buttons.
- A `TextField` with `hintText` (no `labelText`) and `maxLines > 1` doesn't expose the hint via `getByLabel`/`getByPlaceholder`/`getByText` — none of them matched in testing. `page.getByRole('textbox').first()` did. Separately, `maxLines > 1` means Enter inserts a newline instead of firing `onSubmitted` (confirmed: the typed text just sat in the field, nothing sent) — drive the actual send button (`page.getByRole('button').last()` if it's the only other button on the route) rather than `keyboard.press('Enter')`.
- The default `AppBar` back arrow has no visible text but is reachable via `page.getByRole('button', { name: 'Back' })` — the English MaterialLocalizations tooltip, even though the rest of the UI is Turkish (no `localizationsDelegates` configured in `main.dart`).
